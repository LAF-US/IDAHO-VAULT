# BRIEF - LAF-25 Automation and Arborscaping Audit

## Summary

This brief records a Janitor pass over three related surfaces:

- automated and timed triggers
- line-difference and encoding drift
- GitHousekeeping branch sprawl observed in `Screenshot 2026-04-04 092805.jpg`

The current system is active, but not yet cohesive.

- The timed triggers are real and working, but they do not currently converge on one housekeeping doctrine.
- Branch cleanup is too conservative for no-PR agent branches, which allows old branches to accumulate far behind `main`.
- Tag normalization and daily-note normalization are partially scripted, but they still encode legacy assumptions and do not define a clean modern metadata core.
- Text normalization is under-governed: binary assets are governed in `.gitattributes`, but text files are not.

## Automated / Timed Trigger Assessment

### 1. Daily rollover

`/.github/workflows/daily-rollover.yml` is the cleanest timed automation currently visible.

- It runs daily on schedule and on manual dispatch.
- It generates a branch in the form `bot/daily-rollover-YYYY-MM-DD`.
- It opens a pull request only when there are actual staged changes.
- It uses `delete-branch: true`, which means this lane already behaves like a self-pruning automation.

Assessment: functioning and disciplined.

### 2. Janitor sweep

`/.github/workflows/janitor-sweep.yml` is not currently a general janitor.

- It does not perform recurring housekeeping across the vault.
- It only runs when `Daily To-Do Rollover` completes with a `failure` conclusion.
- Its current role is closer to incident reporting than routine maintenance.

Assessment: functioning, but scoped too narrowly to serve as the standing vault janitor.

### 3. Sort audit

`/.github/workflows/sort-audit.yml` runs weekly and writes directly back to the repo.

- It generates audit output under `!/`
- It validates the output
- It commits and pushes directly on schedule

Assessment: functioning, but it bypasses the PR lane that other automations use. This makes the automation garden inconsistent.

### 4. Proposed moves

`/.github/workflows/vault-propose-moves.yml` runs weekly and opens a branch/PR lane if move proposals exist.

- It creates `vault-moves-YYYY-MM-DD`
- It opens a PR based on the generated proposal summary

Assessment: functioning, but it creates another long-lived branch family unless those branches are actively pruned.

### 5. Branch cleanup

`/.github/workflows/branch-cleanup.yml` is the direct source of the branch-garden problem visible in the screenshot.

What it does well:

- deletes agent branches immediately when a PR closes
- deletes branches already merged into `main`
- deletes branches whose PRs are closed

What it does poorly:

- keeps agent branches with no PR at all
- has no age threshold
- has no behind-`main` threshold
- has no weekly census artifact

Assessment: functioning, but under-pruning. The current logic treats "no PR" as "still possibly alive" forever.

## Line Differences / Encoding Drift

The repo has a line-difference policy gap.

### What is governed

`.gitattributes` governs binary/LFS asset classes such as audio, video, images, office files, archives, and databases.

### What is not governed

There is no repo-wide normalization policy for:

- `.md`
- `.py`
- `.yml`
- `.json`
- `.sh`

That means the repo currently lacks a durable shared answer for:

- LF vs. CRLF on text files
- UTF-8 expectations for prose documents
- handling of mojibake or damaged punctuation

### Evidence of drift

- `VAULT-CONVENTIONS.md` currently displays visible mojibake in terminal inspection.
- `normalize_tags.py` preserves existing newline style instead of converging it.
- `tidy_daily_notes.py` hardcodes an older frontmatter order and older daily-note assumptions.

Assessment: the repo is preserving local text habits rather than enforcing a single text doctrine.

- `status/*` for workflow state

## GitHousekeeping Protocol: Arborscaping

The screenshot branch list indicates a garden without a pruning calendar.

### Proposed doctrine

1. Every agent branch must be in one of two states:
   - attached to an open PR
   - explicitly temporary and younger than a set threshold

2. Any agent branch with no PR and older than 7 days becomes prune-eligible.

3. Any agent branch with no PR and more than 100 commits behind `main` becomes priority-prune.

4. Closed-PR branches remain immediate-delete.

5. Merged branches remain immediate-delete.

6. Weekly cleanup should emit a durable census note under `!/` rather than only a transient Actions summary.

### Proposed weekly branch census fields

- branch name
- prefix family
- PR state
- age in days
- commits behind `main`
- recommended action
- deletion reason, if deleted

### Operational judgment

The repo does not currently need more branch creation power.
It needs a stronger and more explicit pruning rule for branches that were never formalized into PR work.
