# Bartimaeus — LAF-40 POSIWID Rubric

**Purpose:** test whether a system is producing output, not merely describing intent.

**Core test:**  
Does the issue terminate in something outside itself?

If a ticket ends only in self-description, it is still a plan.  
If a ticket points to a real artifact, it is behaving like a system.

---

## Acceptable External Terminations

A ticket should terminate in one of these artifact classes:

1. A durable document
2. A runnable or configured integration
3. A produced dataset, tracker, report, or output artifact
4. A verified handoff into another operational system

If it terminates in none of these, treat it as governance vapor.

---

## LAF-40 Test Surfaces

### 1. Daily Cycle (`LAF-31` to `LAF-34`)

**Expected output:**  
Each issue description should contain inherited handoff context from the prior day:

- what was done
- what is next
- what is blocked

**Failure mode:**  
If the descriptions are empty `Todo` shells, the cycle is decorative rather than generative.

### 2. Budget Bill Tracker (`LAF-16`)

**Expected output:**  
A corresponding usable tracking artifact should exist somewhere durable:

- Drive
- Box
- vault
- or another confirmed system of record

**Failure mode:**  
If `LAF-16` only points back to itself or to process language, the tracker is theater rather than tracking.

### 3. Gemini Code Partner (`LAF-23`)

**Expected output:**  
The description should contain operational specifics:

- Gemini's role
- access boundaries
- what it has produced
- what it is expected to do next

**Failure mode:**  
If the issue reads like architecture prose without concrete instructions or outputs, it is vaunted, not vaulted.

### 4. Linear MCP (`LAF-36`)

**Expected output:**  
Setup instructions should be current and complete enough that a fresh Claude Code instance could follow them and connect.

**Failure mode:**  
If the instructions are stale, partial, or aspirational, the integration is not real.

---

## Rubric

For each issue under review, test:

1. **Intent**  
What does the ticket claim the system is for?

2. **Artifact**  
What concrete thing exists because of the ticket?

3. **External termination**  
Does the issue point to something outside itself?

4. **Repeatability**  
Could the output be produced again by following the documented path?

5. **Fresh-agent viability**  
Could a newly awakened agent verify or use the result without oral tradition?

---

## POSIWID Reading Rule

The purpose of the system is what it does.

Therefore:

- if `LAF-31` to `LAF-34` create empty shells, the system's purpose is issue generation, not cycle continuity
- if `LAF-16` produces no tracker artifact, the system's purpose is tracker theater, not tracking
- if `LAF-23` yields only framing language, the system's purpose is role inflation, not partnership
- if `LAF-36` cannot be followed by a fresh agent, the system's purpose is aspirational integration, not working connection

---

## Closing

Bartimaeus should prefer evidence of downstream artifacts over statements of intent.

When in doubt, ask:

**What exists now that did not exist before this issue?**

If the answer is only more description, the system is still speaking to itself.
